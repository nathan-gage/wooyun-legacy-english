# 多租户 SaaS 平台越权访问完整测试计划

> 基于 WooYun 22,132 个真实漏洞案例方法论 | 覆盖水平越权(IDOR)、垂直越权(权限提升)、未授权访问三大维度

---

## 一、测试背景与范围

### 1.1 目标系统画像

| 维度 | 描述 |
|------|------|
| 系统类型 | 多租户 SaaS 平台（REST API） |
| 客户端 | Web 前端 + 移动端 |
| 参与角色 | 匿名用户、普通用户、租户管理员、超级管理员、API 客户端 |

| 信任边界 | 租户间隔离、角色间权限边界、认证/未认证边界 |

### 1.2 WooYun 数据支撑

授权域在 WooYun 数据集中共 **6,838 个案例**，是仅次于认证(8,846)和信息泄露(6,446)的第三大漏洞类别。其中：

| 子类别 | 案例数 | 高危占比 | 本测试计划对应章节 |
|--------|--------|----------|-------------------|
| 水平越权/IDOR | 1,935 | 62.3% | 第三章 |

| 未授权访问/缺失认证 | 3,993 | 58.2%+ | 第五章 |

## 三、水平越权（IDOR）测试计划

> WooYun 案例参考：「北京现代某平台可越权遍历所有用户上传证件（几百万身份证件/行驶证件/发票/驾驶证）」—— 顺序文件 ID 且无所有权检查。

### 3.1 测试原理

IDOR（Insecure Direct Object Reference）的本质：服务器未验证"请求者是否有权访问该资源"。在多租户场景下，存在两层水平越权：

1. **同租户内用户间越权**：用户 A1 访问同租户用户 A2 的私有数据
2. **跨租户越权**：租户 A 的用户访问租户 B 的数据（**多租户 SaaS 最致命的漏洞**）

### 3.2 系统化测试协议

#### 步骤一：识别所有资源标识符

对每个 API 端点，标记资源标识符出现的位置：

| 标识符位置 | 示例 | 风险等级 |
|-----------|------|---------|
| URL 路径 | `/api/v1/users/{user_id}/orders` | 高 |
| 查询参数 | `/api/v1/orders?user_id=123&tenant_id=456` | 高 |
| POST/PUT 请求体 | `{"user_id": 123, "action": "view"}` | 高 |
| HTTP 头 | `X-User-Id: 123` / `X-Tenant-Id: 456` | 高 |
| Cookie | `user_session=base64(user_id=123)` | 中 |

**关键参数清单：** `user_id`, `uid`, `id`, `order_id`, `file_id`, `account_id`, `tenant_id`, `org_id`, `workspace_id`, `doc_id`, `msg_id`, `invoice_id`, `project_id`

#### 步骤二：CRUD 全覆盖测试

对每个包含资源标识符的端点，以用户 A1 的身份登录，替换为用户 A2/B1 的标识符：

```
对于每个端点 + 每项 CRUD 操作：

[C] 创建：
  - [ ] A1 能否为 A2 创建资源？（POST /api/users/A2_id/orders）
  - [ ] 租户A用户能否在租户B下创建资源？（body 中 tenant_id 替换）

[R] 读取（最常见的 IDOR）：
  - [ ] A1 能否查看 A2 的订单/个人信息/文件？
  - [ ] A1 能否查看租户B的数据？
  - [ ] API 响应是否返回了超出 UI 显示范围的字段？（API 过度获取）

[U] 更新：
  - [ ] A1 能否修改 A2 的个人资料/设置？
  - [ ] A1 能否修改租户B的配置？

[D] 删除：
  - [ ] A1 能否删除 A2 的资源？
  - [ ] A1 能否删除租户B的资源？
```

#### 步骤三：标识符操纵技术

> WooYun 教训：「IDOR 有编码绕过、参数污染、JSON 嵌套。简单改 ID 只占案例的 30%。」

| 技术 | 测试方法 | 示例 |
|------|---------|------|
| 直接 ID 替换 | `id=123` → `id=124` | 最基本，但仍有效 |
| ID 枚举/遍历 | 从 1 到 N 遍历 | 顺序整数 ID 时极高危 |
| 参数污染(HPP) | `?uid=123&uid=456` | 服务器可能取最后一个值 |
| 数组注入 | `uid[]=123` 或 `uid=123,456` | 绕过类型检查 |
| JSON 嵌套 | `{"user": {"id": 456}}` | 绕过顶层参数过滤 |
| Base64 编码 | `id=base64(456)` | 替换编码后的 ID |
| 十六进制编码 | `id=0x1c8` | 替换为目标 ID 的 hex |
| 负数/零值 | `id=0`、`id=-1` | 可能返回默认/全部数据 |
| UUID 预测 | 分析 UUID 版本（v1 含时间戳，可预测） | UUID v1 可逆推 |
| 哈希碰撞 | 如果 ID 是短哈希（如 MD5 前 8 位） | 碰撞概率分析 |
| 通配符 | `id=*` 或 `id=all` | 某些实现会返回所有 |

#### 步骤四：多租户隔离专项

```
跨租户 IDOR 测试矩阵：

[ ] 替换请求中的 tenant_id/org_id/workspace_id
[ ] 用租户A的 token 访问租户B的 API 路径（/api/tenants/B_id/...）
[ ] 共享资源测试：文件、报告、仪表板是否有租户级别的访问控制
[ ] 搜索/列表接口是否泄露其他租户的数据
[ ] 导出功能是否能导出其他租户的数据
[ ] 通知/消息系统是否跨租户可见
[ ] Webhook 配置是否能接收其他租户的事件
```

## 四、垂直越权（权限提升）测试计划

> WooYun 数据：255+86 个权限提升案例。「任意账号访问」子类别 86.4% 高危，是所有授权漏洞中危害最大的。

### 4.1 测试原理

垂直越权 = 低权限用户执行了高权限操作。在多租户 SaaS 中，存在三层垂直越权风险：

```
攻击路径：
  普通用户 → 租户管理员（租户级权限提升）
  租户管理员 → 超级管理员（平台级权限提升）
  匿名用户 → 任何已认证角色（认证绕过即权限提升）
```

### 4.2 系统化测试协议

#### 步骤一：映射权限层级与管理端点

```
1. 识别所有权限级别：
   [ ] 匿名（未认证）
   [ ] 已注册普通用户
   [ ] 付费用户/VIP
   [ ] 租户管理员
   [ ] 平台运维
   [ ] 超级管理员

2. 收集管理端点（多源发现）：
   [ ] 直接猜测：/admin/*, /api/admin/*, /manage/*, /internal/*
   [ ] JavaScript 源码分析：搜索 "admin", "manage", "internal", "role" 关键词
   [ ] API 文档/Swagger：查找标记为 admin 或 internal 的端点
   [ ] robots.txt / sitemap.xml：查看 Disallow 的路径
   [ ] 前端路由配置（React Router/Vue Router）中的隐藏路由
   [ ] 移动端反编译：硬编码的管理 API
   [ ] 错误页面泄露的内部路径
```

#### 步骤二：低权限访问高权限端点

用普通用户的 Token/Session 直接请求管理端点：

```
对于每个管理端点：

[ ] 直接访问：用普通用户 Token 请求管理员 API
    GET /api/admin/users         (Authorization: Bearer <user_token>)
    GET /api/admin/tenants       (Authorization: Bearer <user_token>)
    POST /api/admin/settings     (Authorization: Bearer <user_token>)

[ ] HTTP 方法切换：
    原始：GET /api/admin/users → 403
    尝试：POST /api/admin/users → ?
    尝试：PUT /api/admin/users → ?
    尝试：PATCH /api/admin/users → ?
    尝试：DELETE /api/admin/users → ?

[ ] 路径大小写/编码变体（绕过路由级权限检查）：
    /api/admin/users
    /api/Admin/users
    /api/ADMIN/users
    /api/%61dmin/users
    /api/admin/./users
    /api/;/admin/users
    /api/admin;/users
    /api/..;/admin/users
```

#### 步骤三：参数级权限提升

```
注册/更新个人资料时的角色篡改：

[ ] 注册时注入角色参数：
    POST /api/register
    {"username":"evil", "password":"xxx", "role":"admin"}
    {"username":"evil", "password":"xxx", "is_admin":true}
    {"username":"evil", "password":"xxx", "level":9}
    {"username":"evil", "password":"xxx", "type":"superadmin"}
    {"username":"evil", "password":"xxx", "permissions":["admin","write","delete"]}

[ ] 更新个人资料时提升权限：
    PUT /api/users/me
    {"name":"normal_user", "role":"admin"}

[ ] 通过隐藏参数提升：
    在正常请求中添加：group_id, role_id, privilege, admin, is_staff, is_superuser

[ ] 批量赋值（Mass Assignment）：
    发送完整的用户对象（包含角色字段），观察服务器是否过滤
```

#### 步骤四：租户管理员越权到平台管理

```
[ ] 租户管理员能否访问 /api/platform/* 端点
[ ] 租户管理员能否查看/修改其他租户的信息
[ ] 租户管理员能否创建超级管理员账号
[ ] 租户管理员能否修改平台级配置（计费、套餐、全局设置）
[ ] 租户管理员的 JWT/Token 中是否包含可篡改的角色字段
```

### 4.3 JWT/Token 权限提升专项

```
如果系统使用 JWT：

[ ] 解码 JWT payload，检查是否包含 role/permissions 字段
[ ] 修改 role 字段后重新签名（尝试 none 算法、空密钥）
[ ] 修改 tenant_id 字段
[ ] 修改 user_id 字段（JWT 层面的 IDOR）
[ ] 检查 JWT 是否使用弱密钥签名（常见弱密钥字典爆破）
[ ] 检查服务器是否验证 JWT 签名（篡改 payload 但保留原签名）
[ ] 检查 JWT 过期时间是否被服务器强制验证
```

### 4.4 WooYun "任意X" 攻击类别专项

> WooYun 数据：529 个案例，51-86% 高危。与 IDOR 不同，"任意X"是关于执行"你不应该有的操作"。

| 子类别 | 案例数 | 高危占比 | SaaS 场景测试点 |
|--------|--------|----------|----------------|
| **任意账号访问** | 220 | **86.4%** | 无需凭证以任意用户身份登录/操作。测试：能否通过 API 直接以其他用户身份执行操作，不经过正常认证流程 |
| **任意修改** | 159 | 63.5% | 修改任意记录。测试：能否修改其他租户的配置、用户资料、业务数据 |
| **任意用户注册** | 24 | **75.0%** | 绕过注册控制。测试：关闭注册时仍能注册；能否注册为管理员角色；能否绕过邀请码/域名限制 |
| **任意查看** | 45 | 55.6% | 超越 IDOR 的批量数据查看。测试：管理员视图、批量导出、未过滤的列表接口 |
| **任意删除** | 41 | 51.2% | 删除任意记录。测试：能否删除其他租户的用户、数据、配置 |
| **任意操作** | 40 | **72.5%** | 执行特权操作。测试：能否以普通用户身份执行审批、发布、执行等管理操作；能否自我审批 |

**任意X 测试协议：**

```
对于每个写入/删除/管理操作：

1. 识别权限模型
   - 谁应该被允许执行此操作？
   - 权限检查在用户级别、角色级别还是对象级别？

2. 测试权限绕过
   [ ] 以无特权用户身份执行操作
   [ ] 对非当前用户拥有的对象执行操作
   [ ] 批量操作：通过 API 枚举修改/删除全部
   [ ] 以普通用户身份执行仅限管理员的操作（审批、发布、停用账号）
   [ ] 自我审批：创建请求 + 审批自己的请求

3. 测试注册控制
   [ ] 当注册"关闭"或"仅邀请"时能否注册
   [ ] 能否以管理员/提升的角色注册
   [ ] 能否绕过邮箱域名限制（SaaS 常见：只允许 @company.com）
   [ ] 能否绕过注册的手机/邮箱验证
```

### 4.5 WooYun 真实案例参考

| 案例 | 漏洞模式 | 影响 |
|------|---------|------|
| 挖财网权限绕过登录其他用户账号 | 垂直越权 — 账号接管 | 任意用户账号接管 |
| 暴风墨镜某站SQL注入/59张表/权限控制数据库 | 垂直越权 — 注入提权 | 完整数据库访问 |
| 美国东航网站严重订单信息泄漏及权限绕过 | 垂直越权 — 角色绕过 | 订单数据 + 权限提升 |

## 五、未授权访问测试计划

> WooYun 数据：2,102+1,891 个案例。58.2% 的发现 = 管理后台完全暴露在互联网，无任何认证。

### 5.1 测试原理

未授权访问 = 端点完全不检查身份认证，匿名即可访问。这是最基础也是最危险的漏洞——不需要任何账号。

### 5.2 端点发现矩阵

| 发现方法 | 操作步骤 | 目标 |
|---------|---------|------|
| 直接 URL 猜测 | 逐一访问常见管理路径 | `/admin`, `/console`, `/debug`, `/status`, `/api/docs`, `/internal` |
| robots.txt | `GET /robots.txt` 查看 Disallow 路径 | 被"禁止"但未做认证的路径 |
| sitemap.xml | `GET /sitemap.xml` 枚举 URL | 意外暴露的管理 URL |
| JavaScript 源码 | 搜索前端 JS 中的 API 端点 | 硬编码的内部 API 地址 |
| API 文档泄露 | 检查 `/swagger-ui.html`, `/api-docs`, `/openapi.json`, `/redoc` | 完整 API 清单 |
| 错误页面 | 触发 404/500 错误查看堆栈跟踪 | 暴露的内部路径 |
| HTTP OPTIONS | 对各端点发送 OPTIONS 请求 | 暴露允许的方法和端点 |
| 路径遍历 | `/..;/admin`, `/%2e%2e/admin` | 绕过路径级认证 |

### 5.3 系统化测试协议

```
对于每个发现的端点，按以下步骤逐一测试：

步骤 1：完全匿名访问
  [ ] 不带任何 Authorization 头和 Cookie 直接 GET 请求
  [ ] 如果返回 200 → 记录为【严重】未授权访问
  [ ] 如果返回数据 → 评估数据敏感性

步骤 2：绕过重定向
  [ ] 如果 302 重定向到登录页 → 检查响应体是否仍含数据
  [ ] 用 curl -L 跟踪 vs 不跟踪，对比响应
  [ ] 添加 X-Forwarded-For: 127.0.0.1（伪装内网来源）
  [ ] 添加 X-Original-URL / X-Rewrite-URL 头

步骤 3：绕过 403
  [ ] 替代 HTTP 方法：GET → POST → PUT → DELETE → PATCH → HEAD
  [ ] 路径规范化：/admin/ vs /admin vs /Admin vs /ADMIN
  [ ] URL 编码：/%61dmin (a→%61)
  [ ] 双重编码：/%2561dmin
  [ ] 路径穿越：/accessible/../admin
  [ ] 分号注入：/admin;.css（某些中间件会忽略分号后内容）
  [ ] 换行注入：/admin%0a
  [ ] 添加后缀：/admin.json, /admin.html, /admin/

步骤 4：API 版本绕过
  [ ] /api/v1/admin/users → 403
  [ ] /api/v2/admin/users → ?（新版本可能未加权限）
  [ ] /api/internal/admin/users → ?
  [ ] /api/admin/users.json → ?
```

### 5.4 多租户 SaaS 特有的未授权访问点

```
高危未授权端点清单：
```

## 八、漏洞评级标准

### 8.1 严重性评级

| 级别 | 定义 | 示例 |
|------|------|------|
| **严重** | 跨租户数据泄露/篡改；任意账号接管；管理后台完全暴露 | 替换 tenant_id 可访问所有租户数据 |
| **高** | 同租户内 IDOR 泄露敏感信息；普通用户提权至管理员 | 遍历 user_id 获取所有用户个人信息 |
| **中** | 有限范围的越权（如仅能查看不能修改）；非敏感数据泄露 | 能查看其他用户的公开信息 |
| **低** | 需特殊前提条件；影响极有限 | 能看到其他用户的昵称 |

### 8.2 多租户场景的特殊评级规则

```
跨租户越权 → 自动提升一个等级
  - 同租户 IDOR 读取 = 高危
  - 跨租户 IDOR 读取 = 严重
  - 跨租户 IDOR 写入/删除 = 严重（可能影响所有客户）
```

## 十、防御建议总结

> 基于 WooYun 6,838 个授权漏洞案例的修复数据提炼。

### 10.1 代码层面

| 措施 | 说明 | 对应漏洞 |
|------|------|---------|
| **默认拒绝** | 公开端点白名单制，其余一切需认证 | 未授权访问 |
| **所有权验证** | 每个查询验证 `resource.owner_id == current_user.id` | IDOR |
| **ORM 级过滤** | `Model.where(user_id: current_user.id, tenant_id: current_tenant.id)` | IDOR + 多租户隔离 |
| **不可预测 ID** | UUID v4 替代顺序整数 | IDOR 枚举 |
| **RBAC/ABAC** | 基于角色或属性的访问控制框架 | 垂直越权 |
| **职能分离** | 创建者不能审批自己的请求 | 任意操作 |
| **批量赋值防护** | 白名单允许更新的字段，拒绝 role/permission 等 | 参数级提权 |

### 10.2 架构层面

| 措施 | 说明 |
|------|------|
| **API 网关集中授权** | 在网关层统一执行认证和基础授权检查 |
| **多租户数据库隔离** | 所有查询强制 tenant_id 过滤，或使用独立 schema/数据库 |
| **零信任架构** | 每个请求验证身份和权限，无论网络来源 |
| **URL 模式保护** | 框架级路由授权（Spring Security、Django Permissions） |
| **最小化 API 响应** | 仅返回客户端需要的字段，不返回完整数据库对象 |

### 10.3 监控与告警

| 监控项 | 触发条件 |
|--------|---------|
| 跨用户访问模式 | 单个用户访问大量其他用户的资源 |
| 管理端点异常访问 | 非管理员 IP/Token 访问 /admin/* |
| ID 枚举检测 | 单个会话发出顺序 ID 请求 |
| 权限变更审计 | 所有角色/权限修改记录并告警 |
| 批量操作异常 | 单用户短时间内修改/删除大量记录 |
| 跨租户访问 | 任何请求的 tenant_id 与 Token 中的不匹配 |
